Développer des solutions

Concevoir, programmer, automatiser : créer des outils qui industrialisent le traitement de la donnée et répondent à de vrais besoins.

Définition du bloc

Qu'est-ce que « Développer » ?

Le bloc Développer regroupe les compétences qui permettent de passer d'une analyse ponctuelle à une solution pérenne : écrire du code structuré, automatiser des traitements répétitifs, intégrer des données depuis des sources externes, et concevoir des outils que d'autres peuvent utiliser et faire évoluer.

C'est le bloc qui fait le lien entre la science des données et l'ingénierie logicielle. Un analyste qui sait développer ne dépend pas d'une équipe technique pour chaque besoin : il construit lui-même ses pipelines, ses scripts de collecte, ses interfaces de restitution. C'est ce qui rend un profil data vraiment autonome.

Dans le référentiel du BUT Science des Données, ce bloc insiste sur la capacité à répondre à un besoin concret par une solution technique fiable, maintenable et documentée — pas juste un script qui fonctionne une fois.

Compétences

Ce que j'ai développé

Cinq axes de compétences techniques, de la logique algorithmique de base à la conception de pipelines de traitement complets.

🐍 Programmation Python

Python est le langage central de ma formation data. J'ai progressé des scripts linéaires aux programmes structurés : fonctions avec typage, gestion des exceptions, modules réutilisables, et orienté objet (classes, héritage, encapsulation) pour les projets qui le justifient. La question n'est pas "est-ce que ça marche" mais "est-ce que quelqu'un d'autre peut relire, comprendre et modifier ce code".

Sur la SAÉ MonVoyage, j'ai structuré l'application en classes distinctes (collecteur API, parseur HTML, exporteur CSV/JSON) — une architecture qui a rendu le débogage et les tests unitaires nettement plus simples que sur mes premières SAÉ où tout était dans un seul script de 300 lignes.

🔧 Développement d'outils data

Développer un outil, ce n'est pas la même chose qu'écrire un script d'analyse. Un outil est conçu pour être utilisé par quelqu'un d'autre, potentiellement dans des conditions différentes de celles prévues. Ça implique une gestion robuste des erreurs, des messages clairs, une documentation, et une séparation nette entre la logique métier et l'interface.

Sur la SAÉ MonVoyage, l'outil devait fonctionner même si l'API était indisponible sur certaines pages, si une destination n'avait pas tous ses champs renseignés, ou si le format de la réponse changeait légèrement. Anticiper ces cas avant qu'ils surviennent est une compétence de développeur, pas d'analyste.

🌐 Intégration de données (API & scraping)

La collecte automatisée de données demande de gérer bien plus que la requête HTTP : authentification par token, pagination, rate limiting, retry sur timeout, parsing sélectif du HTML, normalisation des données extraites. Chaque source est différente, et la robustesse du code dépend de la façon dont on anticipe ses particularités.

Sur MonVoyage, j'ai combiné deux sources avec des structures radicalement différentes. Le travail de normalisation a représenté autant d'effort que la collecte elle-même — et la logique de correspondance entre les deux sources a demandé une réflexion algorithmique précise (nettoyage de chaînes, comparaison floue de noms).

🏗️ Conception et architecture du code

Bien concevoir un projet avant de coder évite de réécrire deux fois. J'ai appris à structurer un projet Python en dossiers logiques (collecte, traitement, export, tests), à définir des interfaces claires entre les composants, et à rendre chaque module testable indépendamment.

Cette discipline de conception est ce qui permet à un projet de grandir sans devenir ingérable. Sur le Datawarehouse, la séparation entre les scripts de création de schéma, les scripts d'alimentation et les scripts de vérification a permis de modifier un composant sans casser les autres — ce qui n'était pas le cas dans mes premières SAÉ.

🌍
SAÉ — Application Python de collecte

MonVoyage — Application de collecte multi-sources avec génération de carte interactive

Contexte & problématique

L'objectif était de développer une application Python complète et autonome capable de collecter des données sur des destinations de voyage depuis deux sources distinctes (une API REST et un site web par scraping), de les fusionner, de les nettoyer, et de produire deux livrables : un fichier CSV structuré et une carte interactive géolocalisée. L'application devait fonctionner de bout en bout sans intervention manuelle.

Ce que j'ai développé concrètement

  • Classe VoyageCollector : appels API paginés avec authentification token, retry automatique sur timeout, logging des erreurs
  • Classe VoyageScraper : parsing HTML avec BeautifulSoup, sélecteurs CSS robustes, gestion des structures de page non uniformes
  • Module merger.py : réconciliation des deux sources par normalisation des noms (lowercase, suppression accents, distance de Levenshtein pour les correspondances approchées)
  • Module exporter.py : génération CSV avec csv.DictWriter et export JSON
  • Module mapper.py : carte Folium avec marqueurs colorés par catégorie, popups avec résumé des données, export HTML autonome
  • Point d'entrée main.py : orchestration, arguments CLI, configuration

Choix techniques

  • Architecture en classes plutôt qu'en fonctions isolées : facilite le test unitaire de chaque composant indépendamment
  • requests.Session() pour réutiliser la connexion HTTP et les headers sur les appels paginés
  • Logging avec le module logging standard (niveaux DEBUG/INFO/ERROR) plutôt que des print() — permet de désactiver les logs en production
  • Fichier config.py séparé pour les constantes (URL, timeout, noms de colonnes) — évite les valeurs magiques dispersées dans le code

Difficultés rencontrées

  • Le site scrappé modifiait sa structure HTML selon le type de destination : les sélecteurs CSS fallaient pour certaines pages — résolu avec des sélecteurs alternatifs et une vérification de la présence de l'élément avant extraction
  • La réconciliation des noms entre les deux sources était la partie la plus complexe : "Côte d'Azur" vs "cote-d-azur" vs "French Riviera" — la normalisation seule ne suffisait pas, il a fallu implémenter une correspondance floue
  • La pagination de l'API retournait parfois des pages vides avant d'avoir atteint la dernière — condition d'arrêt à affiner pour éviter les requêtes inutiles
Résultat : Une application Python modulaire (~450 lignes réparties en 5 modules) capable de collecter, fusionner et exporter des données sur 800+ destinations en moins de 3 minutes. Le fichier CSV produit est directement exploitable pour l'analyse. La carte Folium permet une exploration visuelle immédiate des destinations par catégorie. L'application gère les erreurs de réseau et les données manquantes sans interruption.

SAÉ — Scripts SQL et alimentation de datawarehouse

Datawarehouse Chinook — Scripts d'alimentation ETL sous Oracle

Contexte & problématique

Après la conception du schéma en étoile (abordé dans le bloc Traiter), il fallait développer l'ensemble des scripts permettant d'alimenter le datawarehouse depuis la base source Chinook. Ce n'est pas une simple copie de données : c'est une transformation structurée qui doit respecter l'ordre des insertions, garantir l'intégrité référentielle, et être rejouable proprement (truncate + reload sans erreur).

Ce que j'ai développé concrètement

  • Script 01_create_schema.sql : création des tables de dimensions et de la table de faits, contraintes de clés étrangères, index sur les colonnes de jointure
  • Scripts 02_load_dim_*.sql : alimentation de chaque dimension (Client, Artiste, Album, Temps) avec génération des clés de substitution (ROWNUM ou séquences Oracle)
  • Script 03_load_facts.sql : alimentation de la table de faits par jointures sur les dimensions — l'ordre d'exécution est critique
  • Script 04_check_integrity.sql : requêtes de vérification post-alimentation (comptages, valeurs nulles, orphelins) pour détecter les anomalies
  • Script 00_reset.sql : truncate dans l'ordre inverse des dépendances pour pouvoir rejouer une alimentation complète proprement

Choix techniques

  • Séquences Oracle (CREATE SEQUENCE) pour les clés de substitution — plus fiable que ROWNUM sur des insertions séquentielles
  • Découpage en fichiers numérotés par ordre d'exécution — convention claire qui évite les erreurs d'ordre lors d'une exécution manuelle ou automatisée
  • Requêtes de vérification intégrées au pipeline : on ne considère pas le chargement terminé tant que les checks ne passent pas
  • Commentaires SQL sur chaque INSERT complexe expliquant la logique de transformation

Difficultés rencontrées

  • La table de faits référence toutes les dimensions — une clé de substitution manquante dans une dimension bloquait l'insertion. Il a fallu comprendre l'ordre exact et gérer les cas où un enregistrement source n'avait pas de correspondance dans la dimension cible
  • La dimension Temps nécessitait de décomposer chaque date en attributs (année, mois, trimestre, semaine) : développement d'une requête de génération de calendrier SQL
  • Oracle SQL*Plus imposait une syntaxe différente de MySQL pour certaines opérations — notamment la gestion des séquences et des fonctions de date
Résultat : Un ensemble de 5 scripts SQL exécutables dans l'ordre, qui créent le schéma, chargent les dimensions, alimentent les faits et vérifient la cohérence. Le datawarehouse est chargé en moins de 2 minutes avec 100% des enregistrements source correctement intégrés. Le script de reset permet de rejouer l'alimentation complète proprement à tout moment — une propriété essentielle pour un pipeline ETL fiable.

Projet personnel — Développement web

Ce portfolio — Site multipage HTML/CSS/JavaScript vanilla

Contexte & problématique

Concevoir et développer ce portfolio de compétences de zéro, sans framework, sans template. L'objectif était de produire un site professionnel, responsive, performant et maintenable, tout en démontrant la capacité à appliquer des compétences de développement web dans un contexte réel. Contrainte principale : zéro dépendance externe, aucun framework CSS ou JS.

Ce que j'ai développé concrètement

  • Architecture multipage avec une feuille de style CSS partagée (~1200 lignes) et des variables CSS pour la cohérence visuelle entre les 5 pages
  • Système de thème par page (couleur d'accent différente par bloc de compétences) via des classes CSS héritées sur <body>
  • JavaScript vanilla : navbar responsive avec burger mobile, IntersectionObserver pour les animations au scroll, effet de frappe, barres de progression animées, compteurs
  • Indicateur de progression de page, bouton retour en haut, smooth scroll sur ancres
  • Responsive design mobile-first avec media queries et grilles CSS adaptatives

Choix techniques

  • Pas de framework : charge plus légère, meilleure maîtrise du résultat, compréhension de ce qui se passe réellement dans le navigateur
  • CSS custom properties (variables) pour les couleurs, espacements et transitions : modification globale en un seul endroit
  • IntersectionObserver pour les animations au scroll plutôt que des événements scroll — plus performant (pas de calculs à chaque frame)
  • Animations CSS avec animation-fill-mode: forwards pour les éléments hero — pas de JavaScript pour la visibilité initiale

Ce que ça m'a appris

  • Le débogage CSS est aussi exigeant que le débogage Python — les règles de priorité et la cascade demandent une rigueur similaire à la logique algorithmique
  • La performance web ne s'improvise pas : les animations GPU-accélérées (transform, opacity) vs les animations qui déclenchent un reflow (hauteur, marges) ont un impact visible
  • Concevoir pour mobile en premier force à hiérarchiser l'information — ce qui améliore aussi le design desktop
Résultat : Ce site — 5 pages HTML, 1 fichier CSS partagé, 1 fichier JavaScript. Entièrement développé à la main, sans générateur ni template. Architecture cohérente, thème par page, animations fluides, responsive sur mobile. Ce projet est à la fois un livrable de portfolio et une démonstration concrète de compétences de développement.

Regard critique

Ce que j'en retire

Ce bloc est celui où je mesure le mieux l'écart entre "coder" et "bien coder". Sur mes premières SAÉ, j'écrivais du code qui fonctionnait dans les conditions où je l'avais testé — et qui tombait dès que quelque chose changeait. La gestion des erreurs était inexistante, la structure était aléatoire, et la documentation nulle. Ça "marchait" mais ce n'était pas du développement.

Sur MonVoyage, j'ai pris du temps pour concevoir l'architecture avant de coder : quelles classes, quelles responsabilités, quelles interfaces entre elles. Résultat : quand le format de réponse de l'API a changé en cours de SAÉ, j'ai modifié un seul module sans toucher au reste. C'est ce que la modularité apporte concrètement — et je ne l'avais pas compris avant de l'expérimenter.

Sur le Datawarehouse, j'ai réalisé que le code SQL peut aussi être bien ou mal structuré. Les requêtes d'alimentation sans commentaires sont aussi illisibles que le code Python sans docstrings. La discipline de documentation s'applique aux deux langages.

✔ Points forts

  • Architecture modulaire : chaque composant a une responsabilité unique et claire
  • Gestion robuste des erreurs : l'application continue de fonctionner face aux cas imprévus
  • Versionnage Git avec commits atomiques et descriptifs
  • Configuration externalisée (pas de valeurs magiques dans le code)
  • Logging structuré plutôt que des print() de debug

↗ Axes de progression

  • Tests unitaires : je ne teste pas encore systématiquement avant de considérer une fonction terminée
  • Performance : mes scripts de collecte ne sont pas optimisés pour la parallélisation (requêtes séquentielles alors que du async serait possible)
  • Documentation : les docstrings sont inégales selon les modules — à uniformiser
  • Déploiement : je n'ai pas encore de pratique des environnements de production (Docker, CI/CD)

Ce que j'améliorerais aujourd'hui : sur MonVoyage, j'aurais utilisé asyncio et aiohttp pour paralléliser les requêtes API et réduire le temps de collecte d'un facteur 5 à 10 sur les pages longues. J'aurais aussi ajouté des tests unitaires sur le module de réconciliation des sources, qui est la partie la plus critique et la plus fragile de l'application.

Sur le Datawarehouse, j'aurais écrit le script d'alimentation en Python avec cx_Oracle plutôt qu'en SQL pur — ce qui aurait permis une gestion d'erreur ligne par ligne avec rollback sélectif, un logging détaillé des insertions, et une rejouabilité plus fine. Le SQL pur est lisible mais moins contrôlable quand on veut du fault-tolerance.

Ce bloc dans le monde data

🔧

Data Engineer

Le Data Engineer construit et maintient les pipelines de données : collecte, transformation, chargement, orchestration. Les compétences de ce bloc — Python structuré, SQL, APIs, architecture — sont exactement ce que ce rôle mobilise au quotidien. C'est le métier le plus directement aligné avec le bloc Développer.

📊

Data Analyst technique

Un Data Analyst qui sait développer automatise ses collectes de données, construit ses propres pipelines de préparation, et produit des outils réutilisables par son équipe. Il est autonome sur l'ensemble de la chaîne et n'est pas bloqué en attente de données propres livrées par quelqu'un d'autre.

⚡

Automatisation

Dans les entreprises, tout ce qui est répétitif doit être automatisé. Les rapports hebdomadaires, les extractions de données, les vérifications de qualité — autant de tâches manuelles coûteuses en temps. Savoir développer ces automatisations est une valeur ajoutée directe et mesurable pour une équipe data.

🏗️

Fiabilité et maintenabilité

En entreprise, le code vit dans le temps. Il est maintenu, modifié, repris par d'autres. Un code illisible ou non testé devient une dette technique. Savoir écrire du code propre, documenté et testable n'est pas un luxe — c'est une exigence professionnelle dans toute équipe qui dure.
